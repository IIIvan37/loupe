# Session — 2026-07-07 — tempo-map (Lot C — closes the tempo plan)

Last piece of the tempo-detection upgrade (plan:
[docs/tempo-detection-plan.md](../tempo-detection-plan.md)): **variable tempo**.
The beat grid was already instant-based (capable of variability since Lot B put
real `beat_this` beats in it); this slice derives the **tempo over time** from it
and makes the read-out follow the playhead. On branch `feat/tempo-map` (off
`main`, Lot B merged as PR #67/#68).

## Done
- **Core domain (TDD, red→green)** in [tempo.ts](../../packages/core/src/domain/tempo.ts):
  - `TempoSegment { fromSeconds, bpm }` / `TempoMap` — steady stretches in order.
  - **`buildTempoMap(grid)`**: walks the gaps between consecutive beats; a gap
    straying beyond ±8 % of the running **median** opens a new segment **only if
    the next gap confirms it** — a lone outlier (a missed beat) is jitter, not a
    tempo change. Segment bpm = 60 / median gap (outlier-proof); the new segment
    anchors at the beat that **opens** the new spacing (DAW tempo-map convention:
    a change governs the spacing after it). `< 2` beats ⇒ empty map.
  - **`tempoAt(map, seconds)`**: the felt bpm at an instant — last segment at or
    before it, the first segment before any beat, `undefined` on an empty map.
  - Property tests (fast-check): a steady grid under ±2 % jitter **never** splits;
    every segment anchors at an instant taken from the grid.
- **Web read-out** ([tempo-panel.tsx](../../packages/web/src/app/tempo/tempo-panel.tsx)):
  when the map has more than one segment the BPM read-out **follows the playhead**
  (`tempoAt`) and a new dim **range read-out** (`tempo.range`, « min–max BPM »)
  appears beside it; a steady track keeps the plain representative BPM display.
  Meter + octave ×2/÷2 controls unchanged.
- **[shell-main.tsx](../../packages/web/src/app/workstation-shell/shell-main.tsx)**
  derives the map from the grid (`useMemo(() => buildTempoMap(grid))`) and passes
  the already-threaded `positionSeconds` down — no new plumbing.
- Public surface: `TempoMap`/`TempoSegment` types + `buildTempoMap`/`tempoAt`
  exported from [index.ts](../../packages/core/src/index.ts).
- Lingui: new id `tempo.range` (catalog extracted).

## Decisions
- **Segments are DERIVED, not persisted** — deviation from the plan's
  « `ProjectTempo.segments` » item, user-confirmed: the map is a pure, deterministic
  function of the already-persisted grid (`buildTempoMap` is a linear pass over a
  few hundred beats — sub-ms, memoised per grid, invisible next to the audio
  decode that dominates project reopening). One source of truth, zero manifest
  change, zero migration.
- **Rupture needs confirmation** (two consecutive deviating gaps) so a missed
  beat never splits the map; **median** everywhere so outliers skew nothing.
- **Segment anchor = the beat opening the new spacing** (not the first beat after
  it) — the gap it opens is already at the new tempo, matching what is felt at a
  playhead inside that gap.
- Tolerance is a domain constant (±8 %), not a parameter — no consumer asks to
  tune it (YAGNI; revisit if a real track fragments).
