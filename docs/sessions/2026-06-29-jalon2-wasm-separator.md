# Session — 2026-06-29 — jalon2-wasm-separator

Slice **J2.2** — real client-side stem separator behind the (already-fixed)
`StemSeparator` port. Branch `feat/jalon2-wasm-separator`.

## Done

- **Core (pure, TDD + property + mutation)** — `domain/segment-plan.ts`:
  `planSegments(total, segmentLength, overlap)` (gap-free overlapping windows,
  last truncated) + `transitionWindow(segmentLength, overlap)` (strictly-positive
  trapezoid, normalised) for weighted overlap-add. Exported from `index.ts`.
- **Two real WASM separator adapters**, selectable via `createSeparator('ggml' |
  'onnx')`, both behind the same port. Default = **GGML**.
  - **GGML (default)** — `demucs.cpp` compiled to WebAssembly (fp16, single-threaded
    SIMD). Engine artifacts `public/demucs/demucs.{js,wasm}` are **committed**
    (no npm/prebuilt exists); regenerated by `scripts/build-demucs.sh` (Docker
    `emscripten/emsdk:3.1.51`, pinned `demucs.cpp` commit, patched with
    `EXPORT_ES6=1` so a Vite **module** worker can load the glue). Adapter:
    `demucs-ggml-model.ts` (model contract + worker protocol), `demucs-ggml-worker.ts`
    (module worker: fetch+blob-import the ESM glue, `_modelInit`/`_modelDemixSegment`
    ABI, C++-driven progress), `demucs-ggml-separator.ts`. Model:
    `Retrobear/demucs.cpp` fp16 ~84 MB, fetched+cached.
  - **ONNX (alternative)** — htdemucs via `onnxruntime-web` (StemSplit fp16 ~166 MB),
    module worker, single-threaded wasm SIMD, JS-side chunk + overlap-add using the
    core `segment-plan`. ORT wasm vendored to `public/ort/` (gitignored) by
    `scripts/copy-ort.mjs` (predev/prebuild).
- Shared helpers: `model-cache.ts` (Cache-API + streamed download), `resample.ts`
  (`OfflineAudioContext` → stereo 44.1 kHz), `stem-layout.ts` (4 stems →
  Voix/Batterie/Basse/Autres). `useSeparation` defaults to `createSeparator()`.
  Removed the J2.1 `stub-separator.ts` (now dead).
- **Browser-verified**: GGML separation runs end-to-end (user-confirmed).

## Not done / remaining

- **Speed**: single-threaded CPU htdemucs is heavy — GGML works and is memory-safe
  but **not faster** than ONNX. The real lever (deferred to a follow-up slice):
  **data-parallel multi-worker** — split the track into N overlapping chunks, run N
  `demucs.cpp` workers in parallel, overlap-add via the core `segment-plan` DSP
  (which the GGML path does not yet use — the C++ segments internally). ~N× on
  multi-core, no COOP/COEP, costs N× model memory.
- **WebGPU is a dead end** for the ONNX model: ORT-web 1.27 can't run htdemucs's
  embedded iSTFT (`ConstantOfShape`) on the GPU, and the GPU bundle's asyncify wasm
  drops the CPU kernel for it.
- Slices J2.3–J2.6 (adaptive detection, mixer, grouping, export) unchanged.

## Code review (high effort) — done, findings addressed

8-angle recall review found no happy-path bug (browser-verified). Fixed: shared
`createWorkerSeparator` (dedups the two adapters + rejects a superseded run's

stall during session creation (no ORT progress hook); a 57-token idiomatic
worker-onmessage clone (jscpd informational).

## Decisions

- **Weights licence**: htdemucs weights are research-only (MUSDB18, non-commercial);
  acceptable for loupe (public, non-commercial) — documented in `jalon-2-plan.md`
  and the adapter model files. Same spirit as the Rubber Band→SoundTouch pivot.
- **GGML over ONNX as default**: lighter memory (fp16, no fp32 upcast / pre-packing),
  no `std::bad_alloc` near the wasm32 ceiling, single-threaded SIMD ⇒ no COOP/COEP
  ⇒ Netlify with no header config. ONNX kept as a selectable alternative.
- **Vite gotchas captured**: a /public file can't be `import()`-ed from source
  (dev guard) — ORT sets only `wasmPaths.wasm` (bundled glue), GGML `fetch`es the
  glue and imports it from a **blob URL**. Classic workers with ES imports break in
  dev ⇒ module workers only.
